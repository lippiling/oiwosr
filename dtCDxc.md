为什么要设置线程数？ CPU 相反，核心数字更慢
因为 ExecutorService 的线程数 ≠ CPU 密集型任务的理想并发数。当任务包含时 I/O、当锁等待或远程调用时，线程会被堵塞，需要更多的线程来维持吞吐。盲目设置为 Runtime.getRuntime().availableProcessors()，会导致大量线程空转，频繁抢占 CPU，激增上下文切换。
实操建议：
；

区分任务类型:纯计算型(如图像编码)可根据核心数量设置；混合型(如 Web 请求处理。压力测试后需要调整
用 jstack 或 jcmd &lt;pid&gt; VM.native_memory summary 若，观察线程状态 WAITING 或 BLOCKED 比例高，说明线程闲置多，可以适当增加
避免硬编码核心数，改用配置项 + 例如，启动参数控制 -Dpool.core.size=8


ThreadPoolExecutor 的 keepAliveTime 如何设置不拖慢切换
keepAliveTime 控制空闲线程的存活时间。设置过短(如 10ms)，频繁销毁和重建线程，触发 JVM 线程栈分配/回收及 OS 等级调度费用；设置得太长(例如) 5min)，空闲线程长期占用资源，占用其他任务调度机会。
实操建议：
；

设置短期突发流量场景， keepAliveTime = 60L 秒，配合 allowCoreThreadTimeOut(true)，所有线程都可以超时退出
对于稳定的长连接服务(如 gRPC server），保持 keepAliveTime 为 0(即核心线程永驻)，但必须限制最大线程数，防止 OOM
注意单位：TimeUnit.SECONDS 和 TimeUnit.MILLISECONDS 混合是一种常见的错误，new ThreadPoolExecutor(4, 8, 60, TimeUnit.SECONDS, ...) 是常规写法

使用 ForkJoinPool 代替 ExecutorService 上下文切换会减少吗？
不一定。ForkJoinPool 基于工作盗窃（work-stealing），理论上，单线程可以动态处理多个子任务，降低线程总数的需求。但它只适用于可分解的 CPU 密集型任务(如合并排序、树遍历)有效；带锁，I/O 或者外部依赖的任务，由于任务分割+合并而引入额外费用，窃取逻辑本身也消耗 CPU。


实操建议：
；

仅在满足以下条件时使用 ForkJoinPool：任务无共享状态，可递归分割，执行时间 > 10ms
避免在 compute() 中调用 Thread.sleep() 或 Object.wait()，这会让整个 worker 线程悬挂，影响全局盗窃效率
默认构造的 ForkJoinPool() 使用 availableProcessors() 作为并行度，但实际上应该设置为 Math.min(32, availableProcessors() + 1)，过度并发防小核设备

局部线程变量 ThreadLocal 能避免上下文切换吗？
不能。ThreadLocal 它是一种内存隔离机制，而不是一种调度优化手段。它解决了与上下文切换无关的数据共享冲突。滥用 ThreadLocal 反而加重 GC 压力(尤其是在线程池中) remove()），间接拉高 STW 时间，整体反应恶化。
实操建议：
；

只在真正需要“每行程一份副本”的场景中使用，如事务 ID、不要用来缓存用户上下文的大对象
在线程池环境下，任务结束前必须调用 threadLocal.remove()，否则可能会导致内存泄漏——这是最常被忽视的一点
不要用 ThreadLocal&lt;SimpleDateFormat&gt; 缓存这种非线程安全对象，SimpleDateFormat 本身已过时，应重用 DateTimeFormatter


上下文切换费用从来不能靠“配一个数字”来解决。它隐藏在任务结构、锁粒度、I/O 阻塞点也隐藏在你忘记的地方 remove() 的那行 ThreadLocal 后面。
