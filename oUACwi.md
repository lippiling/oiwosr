沽迷张澄燃


  
Java异常系统通过checked/unchecked分层强制区分外部风险和代码缺陷:ioexception等checked异常必须显式处理，nullpointerexception等unchecked异常必须通过防御编程进行预防；eror不能捕获，自定义异常必须根据业务响应选择继承exception或runtimexception，并使用cause链传递根源。

Java异常系统的初衷是用编译器强制+语义分层，将“必须处理的外部风险”和“应修复的代码缺陷”完全分开。


为什么要把 Exception 分成 checked 和 unchecked 两类
这不是为了增加复杂性，而是为了在编译期划清责任界限：


IOException、SQLException 这类 checked 异常表示程序逻辑没有问题，但外部环境可能有问题（如磁盘断裂、网络断裂）。Java 要求你必须 try-catch 或 throws，否则，编译失败——这迫使你面对依赖的不确定性。

NullPointerException、ArrayIndexOutOfBoundsException 这类 unchecked 异常，继承自己 RuntimeException，编译器不在乎。因为几乎都是写错代码造成的，比如忘记判空，下标硬写。 list.get(100)。不强制捕获是希望你使用防御性编程(比如 Objects.requireNonNull()、集合 size 验证)提前阻止，而不是依靠提前阻止 try 块兜底。
混淆这两类是新手最常踩的坑:对 NullPointerException 写满 catch 块，却对 FileNotFoundException 不管——结果是掩盖 bug，同时也使关键错误被忽视。

为什么 Error 几乎不应该有类型 catch
OutOfMemoryError、StackOverflowError 这些都不是“异常” JVM 已经撑不住的信号:

它们发生时，它们发生时，JVM 可能已经处于不一致状态(如堆内存完全爆炸，甚至是新的异常对象) new 不出来)，这个时候什么都没有 catch 都不可靠。
试图 catch(Error) 并继续运行，可能导致更奇怪的行为（如数据混乱、线程卡死），远不如让过程快速失败，触发监控报警安全。
真正该做的是:用 -Xmx 调整堆的大小和用途 jstack 查栈溢出根源，使用 Arthas 动态诊断—而不是写作 catch 块假装可以处理。

异常的自定义应该继承 Exception 还是 RuntimeException
这取决于你扔掉这个异常的意图：
			
		
；

如果业务规则被破坏，调用人**必须响应**(如支付金额为负，下游系统必须进行退款流程)才能继承 Exception。通过这种方式，编译器将强制对上层代码进行处理，以避免错过关键分支。
如果只是参数明显违法(如引入) null ID 查询用户)，属于调用人使用错误，继承 RuntimeException。这样既保留了语义(说明是调用方的问题)，又不污染正常流程 try-catch 结构。
尽量省事全用 RuntimeException——这将使本应显式处理的业务异常变得沉默和崩溃；不要滥用它 Exception——比如手机号格式的纯输入检查，强制上层处理，增加了不必要的负担。

最容易被忽视的一点:Throwable 在结构方法中，cause 参数不是装饰。链式异常（new ServiceException(下单失败”， e)）原始原因(如数据库连接加班)可以完全透露，否则日志中只剩下一层包装，调查时只能猜测。
