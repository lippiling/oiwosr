懊亓话固把


  
Mybatis事务不生效的主要原因是@transactional不作用于public方法，因为spring AOP代理只拦截public方法；还需要检查@MapperScan配置，避免直接调用同类，合理选择REQUIRED或REQUIRES_等传播行为NEW。

Mybatis事务不生效。让我们看看@transactional是否在public方法中工作
Spring 默认情况下，AOP代理拦截public方法，如果把@Transactional加在private、protected或者在包级方法上，事务根本不会开始——连日志都不会打。这不是Mybatis的问题，而是Spring代理机制的硬限制。
实操建议：

检查目标方法是否为public，且未被final修饰
避免在同一类中调用此类。@Transactional方法（比如serviceA.method1()内部直接调this.method2()），这将绕过代理，事务失败
如果必须同类调用，考虑注入自己的Bean：@Autowired private ServiceA self;，再用self.method2()


Propagation.REQUIRED vs Propagation.REQUIRES_NEW：嵌套调用时，行为差异很大
Propagation.REQUIRED(默认)重用当前事务，不重建；Propagation.REQUIRES_NEW总是挂起当前事务，开新事务。两者在异常回滚、数据库连接、日志一致性等方面表现完全不同。
常见错误现象：

用REQUIRED内部方法抛出异常，外部方法没有捕获，整个事务回滚——但你可能只是想回滚内部方法
用REQUIRES_NEW内部事务提交后，外部事务仍可回滚，导致“已提交但消失”的数据幻觉
内层REQUIRES_NEW外层刚insert但没有comit的数据在方法中找不到(隔离级影响)

示例场景：订单创建后发出通知，通知失败不得导致订单回滚：
@Transactional(propagation = Propagation.REQUIRED)
public void createOrder(Order order) {
orderMapper.insert(order);
notifyService.sendAsync(order); // REQUIRES_NEW更安全
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void sendAsync(Order order) {
// 即使这里有异常，订单数据不受影响
notificationMapper.insert(new Notification(order));
}

执行SQL后，MyBatis没有提交任何事务？检查是否错过了@MapperScan或SqlSession生命周期？
Spring管理事务，但MyBatis操作需要Spring托管SqlSession只能参与事务。如果手动new。 SqlSession，或者Mapper接口未被接受@MapperScan扫描后，SQL将采用自动提交模式（auto-commit=true），@Transactional完全无效。


实操建议：

确认配置类或启动类@MapperScan(&quot;com.example.mapper&quot;)，准确匹配路径
不要在代码里写sqlSessionFactory.openSession()，全部使用@Mapper接口注入
检查Mapper接口方法是否返回void或者受影响行数-返回行数List或Optional不影响事务，但如果误配ResultMap查询失败可能会掩盖事务问题

Propagation.SUPPORTS和Propagation.NOT_SUPPORTED：读多写少场景下的轻量选择
这两种沟通行为经常被忽视，但在高频查询、缓存预热、日志记录等阅读场景中，事务成本可以显著降低。他们不强制开放事务，以避免不必要的连接占用和锁定竞争。
关键区别：


SUPPORTS：如果有事务，就加入，如果没有事务，就不是事务执行——适用于“可有可无”的阅读操作，如统计报表预加载

NOT_SUPPORTED：主动挂起当前事务，以非事务方式执行——适用于写本地日志文件、调用外部HTTP接口等不能进入事务的操作
两者都要求方法内不能出现INSERT/UPDATE/DELETE，否则可能报Cannot execute update in a read-only transaction


注：MySQL默认隔离等级REPEATABLE READ，即使NOT_SUPPORTED，查询仍然可能触发间隙锁——这不是传播行为的问题，而是存储引擎层面的机制。

事务传播不是开关，而是上下文协商。Propagation参数选择错误，从逻辑混乱到数据不一致。最常被忽视的是事务边界是否真的与SQL执行路径对齐——不要只盯着注释，看它是否落在执行SQL的对象实例上。
