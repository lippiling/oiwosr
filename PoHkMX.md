module-info.java 文件必须放在模块根目录中
Java 模块系统不会自动发现或扫描子包中的模块 module-info.java，它只识别顶层源码根目录(即) src/main/java 或 src 下面直接放)下面的文件。放错位置会导致编译器完全忽略模块声明，也可以报告 error: module not found 或者默默退化为非模块化操作。

正确路径：src/main/java/module-info.java

错误路径：src/main/java/com/example/module-info.java(会被视为普通类)
IDE 有时它会自动生成到错误的位置，特别是当旧项目升级时，一定要手动检查并切割到根目录
Maven 构建时若用 maven-compiler-plugin，需显式设 &lt;release&gt;9&lt;/release&gt; 或更高，否则 module-info.java 可能被跳过

requires 和 requires static 区别不在于“是否使用”，而在于“编译期是否依赖于传输”
requires 声明是运行时必要的模块；requires static 这意味着该模块仅在编译过程中需要(例如注释处理器)Lombok），不参与运行时的链接。混淆两者，从启动失败到模块图分析冲突。

用 @NonNull（来自 javax.annotation）做空检查？→ requires static java.annotation

用 HttpClient 发请求？→ requires java.net.http（JDK 11+）
如果误写成 requires static java.net.http，运行时会抛 NoClassDefFoundError

模块图验证（java --list-modules 或 jdeps --module-path）可以帮你确认实际依赖链是否符合预期

exports 和 opens 边界控制非常严格，反射和序列化容易崩溃
exports 控制的是 public 其他模块可以访问类；opens 允许反射访问(包括字段、方法和结构器)，并且只对指定的包生效。不添加 opens 却用 ObjectMapper 或 Spring 注入私有字段？直接 IllegalAccessException。

希望其他模块 new 你的 public class Config？→ exports com.example.config

还想让 Jackson 它的反序列化 private 字段？→ opens com.example.config（不能只靠 exports）
如果只 opens 不 exports，外部模块甚至看不到类名，反射也无从谈起
JDK 9+ 的 --add-opens 是临时绕过手段，上线前一定要补上 opens 声明，否则模块封装将是徒劳的

模块路径（--module-path）和类路径（-cp）不能混用
一旦用了 --module-path，JVM 此时进入“模块模式” -cp 上的 JAR 默认情况下，它成为匿名模块（unnamed module），它可以读取所有命名模块，但另一方面，命名模块默认不能读取匿名模块中的类别。常见症状：明显 requires 写，但提示 package not visible。


；

启动命令必须统一，或全模块路径（java --module-path mods -m my.module/com.example.Main），要么回归传统 classpath
第三方库(如 Log4j、Guava）如果没有模块化，可以打包成自己的模块（Automatic-Module-Name MANIFEST 属性可以辅助识别模块名)，或使用 --add-modules 显式引入
Maven 中 maven-jar-plugin 要配 &lt;archive&gt;&lt;manifestEntries&gt;&lt;Automatic-Module-Name&gt;com.example.lib&lt;/Automatic-Module-Name&gt;&lt;/manifestEntries&gt;&lt;/archive&gt;，否则 jar 名称中的数字/下划线将生成非法模块名

模块系统的真正复杂点不是语法，而是“可见性是分层叠加的”：exports 控制编译期引用，opens 控制运行过程中的反射，requires 控制链接时的可达性， --module-path 它决定了整个模块图的起点。错过任何一层，性能可能是相同的错误——找不到，但原因完全不同。
