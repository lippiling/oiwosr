撬倒党该咎


  
&lt;p&gt;Runtime.getRuntime().freeMemory() 回到JVM堆中已分配但未被对象占用的字节数，直接受GC影响，不反映真实可用内存；used = totalMemory() - freeMemory() 是目前已使用的堆内存估算值。&lt;/p&gt;

Runtime.getRuntime().freeMemory() 返回到底是什么？
它不是“空闲内存”，而是“空闲内存” JVM 堆中未使用的字节数**已分配，但尚未被对象占用。这个值将受到影响 GC 直接影响行为——GC 一跑，freeMemory() 可能瞬间变大；GC 即使堆里满是垃圾，也可能很小，没有触发。

不要用它来判断“内存是否足够”。它不反映真实的可用空间（如未分配的堆叠上限）
它和 totalMemory() 是配套观看：used = totalMemory() - freeMemory() 目前已使用的堆内存估计值
若刚开始应用就调用，freeMemory() 可能接近 totalMemory()，但这并不意味着堆没有压力——只是对象分配不太多

为什么 Runtime.maxMemory() 有时比 -Xmx 设置值小
maxMemory() 返回的是 JVM **实际申请到操作系统的最大内存上限**，不是你写的 -Xmx2g 的字面值。JVM 启动时，将为元空间、线程栈、线程栈预留一部分空间JIT 编译缓存等，因此堆可用的上限会被动态降低。

容器环境常见：cgroup 内存限制低于 -Xmx 时，JVM 会自动降低 maxMemory() 值得适应，避免 OOM Killer 杀进程
Java 8u191+ 和 Java 10+ 默认启用 +UseContainerSupport，此时 maxMemory() 优先考虑容器限制，而不是容器限制 JVM 参数
验证方法：启动时添加 -XX:+PrintGCDetails，看 GC 日志里 “max heap” 实际值是否与 Runtime.getRuntime().maxMemory() 一致

用 Runtime 获取内存指标的三个典型误用场景
这些写作方法看起来合理，但实际上监测失真或干扰 GC：

在高频循环中反复调用 freeMemory() + totalMemory() 计算使用率-频频触发 GC 相反，尝试提高延迟
把 maxMemory() 将其与“系统总内存”进行比较 /proc/meminfo，忽略了 native memory（如 DirectByteBuffer、JNI、JVM 自己的费用完全不计入) Runtime 统计
用 freeMemory() &lt; 1024 * 1024 内存报警阈值-不考虑 GC 触发机会，可能每秒抖动几十次，报警轰炸

假如真的要监控堆状态，java.lang.management.MemoryUsage 提供带时间的上下文，GC 友好的快照，不受影响 GC 中间干扰：中间干扰：

import java.lang.management.ManagementFactory;
import java.lang.management.MemoryUsage;
&lt;p&gt;MemoryUsage heap = ManagementFactory.getMemoryMXBean()
.getHeapMemoryUsage();
long used = heap.getUsed(); // 目前已使用的字节(含) GC 暂存对象）
