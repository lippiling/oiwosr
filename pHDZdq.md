import用于导入类而不是包，import java.util.*只让编译器在java.在util中找到未限名类；import static直接将静态成员引入作用域，提高可读性需要权衡清晰度和命名冲突的风险。

import 它用于导入类，而不是导入包
很多人以为 import java.util.* 这是一个“导入整个包”。事实上，它只是告诉编译器：当代码中出现未限定名的类别（如 ArrayList），请去 java.util 在里面找。实际生效的还是具体的类别，不是包本身。
常见错误现象：import java.util; 编译报错 —— 因为 import 后面必须与类名或通配符匹配 *，不能只写包名。


import java.util.ArrayList;：只导入单个类别，推荐用于明确依赖的场景

import java.util.*;：将**已编译的所有公共类别(不包括子包)导入包中，但不会提高性能，只会影响编译期的分析
重复 import 同一类(如两个不同的包) Date）会导致编译错误，必须用全限定名显式解决

import static 它是为了节省类名前缀，而不是为了“引入静态方法”本身
import static 其实质是将某一类的静态成员(字段或方法)直接拉入当前的作用域，这样你就可以写作了。 assertEquals(1, a) 而不是 Assert.assertEquals(1, a)。但它不会改变语义，也不会影响运行时的行为。
容易踩的坑:
立即学习"Java免费学习笔记(深入)；

过度使用 import static org.junit.Assert.* 代码很难区分哪些是静态方法，哪些是本地变量 —— 比如 fail() 看起来像普通函数调用，其实是 Assert.fail()


import static java.lang.Math.*; 后，sqrt(4) 可以直接使用，但如果同时使用， import static java.util.Objects.*;，本地变量也被定义 sqrt，就会冲突
不能 import static 构造器（即 new XXX() 不受其影响)

该用什么时候？ import static？查看调用频率和上下文的清晰度
适用于该领域公认的高频、无歧义、静态成员。JUnit 的断言、Math 的常量、Collections 工具方法是典型场景；除非团队共识明确，否则谨慎使用业务逻辑中自定义的静态工具方法。


实操建议：

常用于试验类 import static org.junit.Assert.*; 或更细的粒度 import static org.junit.Assert.assertEquals;

数学计算密集型代码可以考虑 import static java.lang.Math.PI; 和 import static java.lang.Math.sqrt;，但避免 import static java.lang.Math.*;

永远不要 import static 一般名称的静态字段(例如 VALUE、TYPE )，容易造成命名污染

IDE 默认情况下，自动导包无法打开 import static，这是有道理的
多数 IDE（IntelliJ / Eclipse）默认只做普通 import，需要手动打开或按快捷键触发 import static。这不是缺乏功能，而是设计克制 —— 静态导入会削弱代码的可读性边界。
隐藏的依赖真的很难处理：例如，你使用了它 Stream.of()，IDE 自动加了 import java.util.stream.Stream;，但是如果你以后再写一遍 of(1,2,3) 并启用了 import static java.util.stream.Stream.of;，当别人读代码时，他们必须倒查才知道 of 来自哪里。
所以，即使节省了键盘输入，也要多问一句:这个静态成员的来源在当前文件中一目了然吗？
