Java泛型是编译期的伪泛型，运行时只能通过编译检查和隐式转换来保证安全；原始类型、反射、@Supresswarnings或通配符误用会绕过检查，导致Classcastexception在运行过程中发生。

Java泛型在编译期间进行了类型检查，并且在运行过程中被擦除，因此它不能阻止所有类型的错误，例如通过反射或原始类型绕过检查，ClassCastException在运行过程中仍有可能抛出。

编译期发生了泛型类型检查
Java泛型是“伪泛型”，底层是通过类型擦除来实现的。编译器可以看到 List，会检查所有 add()、get() 调用是否符合要求 String 将隐式类型转换插入约束和插入 get(0) 返回值自动转换 String）。
但擦除后，JVM 实际只认 List(原始类型)，不保留泛型信息。

使用 javap -c 查看字节码，可以看到 checkcast 自动插入指令 get() 后面
若用 @SuppressWarnings("unchecked") 强制绕过编译检查，或使用反射调用 add() 插入非 String 对象在运行过程中暴露问题
泛型不能限制数组的创建（new T[10] 非法)，因为类型信息在运行过程中不存在

原始类型（raw type）是泛型安全最大的破口
编译器一旦将泛型集合赋予原始类型变量，就会放弃所有类型的检查：
；
			
		
List list = new ArrayList();
List raw = list; // 通过编译，但很危险
raw.add(123);// 编译通过！实际存入 Integer
String s = list.get(0); // 运行时报 ClassCastException
这种写法在遗留代码或误用中很常见 API(例如，一些旧框架返回 List 而非 List）。

始终避免显式声明原始类型的变量(如 List raw）
启用 -Xlint:unchecked 编译选项，让编译器警告原始类型的使用
IDE（如 IntelliJ）默认使用高亮原始类型，不要忽视这些提示

通配符和边界如何影响类型安全？
? extends T 和 ? super T 不是“更松”的泛型，而是准确限制了阅读/写作能力，反而提高了类型安全性:


List extends Number>：可安全读出 Number 及其子类，但不能 add() 任何特定类型(除外) null），防止破坏协变一致性

List super Integer>：可安全写入 Integer 或其子类，但读出只能作为 Object 处理
错用 add() 在 extends 通配符(如 list.add(new Double(1.0))）它将被编译器直接拒绝

真正容易被忽视的是泛型方法中类型参数的推断失效场景——例如，引入 null 或者泛型擦除后的数组会导致编译器无法推导 T，然后放松检查。此时，ClassCastException 藏在看似干净的代码后面。
