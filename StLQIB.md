Runnable 描述“要做什么”，Thread 负责“用什么来执行”；Runnable 无 start() 方法，仅 Thread 可以启动线程并对其进行回调 run()；推荐优先使用 Runnable 避免单继承限制，支持复用和线程池。

Runnable 是任务，Thread 是线程载体
直接得出结论：Runnable 描述「要做什么」，Thread 负责「用什么来执行？」。你不能调用 runnable.start() —— 根本没有这种方法；只有 Thread 实例才有 start()，而且，它的内部最终会被调用到输入中 Runnable.run()(即使你继承了 Thread，run() 也是被 start0() 这个 native 间接触发方法)。
常见错误现象：
- 把 myRunnable.run() 作为启动线程 —— 实际上只是普通方法的调用，仍在主线程中执行，完全没有并发；
- 继承 Thread 后又试图 new 多个例子共享状态(如票数)，结果是每个例子 Thread 子类对象都有独立的份额 total 字段根本不是“共享资源”，只是看起来像。

为什么推荐优先使用？ Runnable？不是因为 Thread 不能共享，但自然不适合
关键不在于“能不能”，而在于“要不要为了一个线程功能而把唯一的继承权放进去”。Java 如果您的业务类别已经继承，只允许单一继承 Service、Fragment 或者框架基类，然后想要增加更多的线程能力，Runnable 是唯一干净的选择。
实操建议：
- ✅ 用 Runnable：任务逻辑需要重用(如多个线程跑同一段计算)、要进线程池（ExecutorService.submit(runnable)）、其他父类已继承；
- ⚠️ 慎用 Thread 子类：只适用于极简主义 demo 或者调试场景，例如暂时开始一个线程打日志，发心跳；
- ❌ 避免混合：不要写 class MyTask extends Thread implements Runnable —— 冗余且容易混淆，Thread 本身已实现 Runnable。
共享资源的真相：和 Runnable/Thread 无关，只与对象的实例有关
网上常说“Runnable 资源可以共享，Thread 不，这是误导。真正决定是否分享的是「同一对象的实例是否操作多个线程？」：public class TicketSeller implements Runnable {
private int total = 10; // 实例变量
public void run() {
synchronized(this) { // 锁定当前实例
if (total > 0) System.out.println("卖出:"" + total--);
}
}
}
// ✅ 共享:两个线程共用同一个线程 TicketSeller 实例
TicketSeller seller = new TicketSeller();
new Thread(seller, "窗口1").start();
new Thread(seller, "窗口2").start();// ❌ 不分享：每一个 Thread 子类是新对象，total 各自一份
class MyThread extends Thread {
private int total = 10;
public void run() { / ... / }
}
new MyThread().start(); // total=10
new MyThread().start(); // total=10(另一份)Thread 构造器里传 Runnable，到底发生了什么？
new Thread(runnable) 并不是“把 Runnable 塞进 Thread”，而是让 Thread 的 run() 内部委托方法 runnable.run()。看看源码就知道：Thread.run() 默认实现是：
			
		
；public void run() {
if (target != null) {
target.run(); // target 是你传进来的 Runnable
}
}
所以你写 new Thread(() -> System.out.println("hi")).start()，本质上还是靠的 Thread 对象启动 OS 回调你的线程 lambda。这层解耦，正是 Runnable 任务逻辑可测试、可替换、可序列化(如发送到远程节点执行)的价值 Thread 只是 JVM 一次性执行环境。
最容易被忽视的一点:Thread 类别本身有大量的状态字段（threadLocals、inheritableThreadLocals、threadStatus 等），频繁 new Thread 会带来额外的内存费用； Runnable 例子要轻得多，配合线程池是生产环境的标准姿势。
