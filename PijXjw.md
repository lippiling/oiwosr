Stream API 它是一种抽象的声明、不可变的、支持并行的数据处理，不修改原始收集或存储数据，只描述操作；适用于一次性链转换、中等数据量、延迟或平行场景，不适用于重复遍历、极简操作、极端性能或break/continue场景。

Java 的 Stream API 它不是用来取代集合操作的“新语法糖”，而是为**声明、不可变、支持并行**的数据处理提供统一抽象。它不修改原始集合或存储数据，只描述“该做什么”，这必须首先明确。
该用什么时候？ Stream，什么时候不应该用？
适合用 Stream 场景:一次性链转换(如过滤+映射+聚合)、逻辑清晰，数据量中等（10^4 ~ 10^6）、延迟执行或并行加速是必要的。
不合适的场景：
- 同一数据需要经常重复同一数据（Stream 只能消费一次)
- 操作非常简单(如单次操作) for 循环取最大值)，使用 Stream 相反，增加开销
- 对性能极其敏感，已知集合非常小（JVM 传统循环优化后可能会更快。
- 需要中途 break/continue 控制流（Stream 没有原生的支持，必须依靠 anyMatch/findFirst 曲线救国）

filter + map + collect 是最常用的组合
这是大多数业务中清洁和组装数据的主要过程。注意三者的顺序不能混乱，中间的每一步都会回到新的 Stream，不要改变原始集合。
常见错误：
- 在 filter 中写副作用逻辑(如 System.out.println()），应改用 peek()(仅用于调试)
- 忘记终端操作(如遗漏) collect() 或 forEach()），不会执行整个流程
- 用 collect(Collectors.toList()) 后来想修改返回的 List，结果抛 UnsupportedOperationException(一些收集器返回不可变容器)List names = users.stream()
.filter(u -> u.getAge() >= 18)
.map(User::getName)
.filter(Objects::nonNull)
.map(String::toUpperCase)
.collect(Collectors.toList());
Stream.iterate 和 Stream.generate 容易误用
这两种方法都是创建无限流的工厂方法，但是如果没有限制直接 collect，会 OOM。必须配合 limit() 或短路终端操作(如 findAny()）。
；
			
		
iterate 适用于有明确起始和迭代规则的序列(如斐波那契、步进数列)；generate 更适合无状态随机值或对象结构。


Stream.iterate(0, n -> n + 2).limit(5) → [0, 2, 4, 6, 8]


Stream.generate(UUID::randomUUID).limit(3) → 三个随机 UUID
错例：Stream.iterate(1, i -> i * 2).collect(Collectors.toList()) → 死循环

并行流（parallelStream()）不是银弹
它底层用 ForkJoinPool.commonPool()，默认线程数等于 CPU 核心数减一。优点是大集合计算密集型任务可以加速；缺点是:
- 非线程安全操作(如往常 ArrayList 里 add）导致数据丢失或异常
- 小集合或 I/O 相反，密集型任务更慢(线程调度费用) > 计算收益）
- 调试困难(堆栈混乱，执行顺序不确定)
真正安全的并行操作:只读数据 + 无状态中间操作 + 使用线程安全的收集器(如 Collectors.toConcurrentMap）List result = numbers.parallelStream()
.map(n -> n * n)
.filter(n -> n % 2 == 0)
.collect(Collectors.toList()); // 这里 collect 是线程安全流动的真正复杂点不是语法，而是理解「延迟执行」「一次性消费」「无状态函数」这三种约束。写完一段流程操作后，首先问问自己：它是否可读、可测量、可中断，不依赖外部状态——如果任何一个都是否定的，你应该考虑改变传统循环或拆卸多个小流程。
