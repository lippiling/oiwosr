HashMap put过程:先计算hash，确定下标，空直接插入；非空先对equals进行比较，覆盖value等于处理哈希冲突；key为null固定放置索引0；自定义key必须重写hashCode和equals并保持一致；修改影响hashCode的字段会导致get失败；链表转红黑树需要同时满足桶内长度≥8且数组长度≥64；扩展触发条件为size&gt;threshold，新容量翻倍，rehash全量；get平均O(1)，最坏O(log n)(红黑树)或O(n)（链表）。

HashMap 的 put 这个过程发生了什么
往 HashMap 在里面放一个键值是对的，而不是简单的计算 hash 然后插入数组。它实际上会走一整套判断链:先算 hash(key)，然后使用该值和数组长度进行和计算以获得下标；如果位置为空，直接创建新位置 Node 放进去；如果不是空的，首先要比较 key 是否相等（equals()），相等就覆盖 value；在考虑哈希冲突处理之前，不平等。
常见错误现象：put 后来拿不到刚存的值，大概率是 key 的 hashCode() 和 equals() 没有写一致——比如只重写 hashCode()，或者两种方法之间的逻辑矛盾。

自定义类作 key 同时，必须同时重写 hashCode() 和 equals()，并保证“相等” key 一定有相同 hash 值”

key 为 null 它总是被放在数组索引中 0 只允许一个位置 null key
不要在 key 对象存储后的修改影响 hashCode() 字段，否则后续 get() 找不到-因为 hash 值变了，查的不是原来的桶

链表转红黑树的阈值和触发条件
当桶内链表的长度达到时 8，且整个 HashMap 的数组长度 ≥ 64 只有这样，链表才会变成红黑树。这两种条件是必不可少的。
为什么不“只要链表足够长就转”？因为在小容量数组下，链表的长度更大，而不是大量的哈希冲突；盲目转树会增加维护费用。数组 ≥ 64 这意味着哈希的分布应该相对均匀，而且长度仍然存在 ≥ 8 链表，更有可能 key 的 hashCode() 实现不合理或数据本身存在偏差。
；

可通过 -XX:AutoBoxCacheMax=200 等 JVM 间接影响参数 Integer 等待缓存对象 hash 分布，但不建议这样调整
如果树化频繁触发，优先考虑 key 类型的 hashCode() 是否过于集中(如只返回固定值或低比特全零)
红黑树退化回链表的阈值是 6，不是 7 或 8 一半-这是为了防止 7 附近反复树化/退化引起抖动

扩展机制如何影响性能和线程安全？
HashMap 在 size &gt; threshold（即 capacity × loadFactor）当触发扩容时，新容量是原来的 2 重新计算所有现有元素的位置。这个过程是全部的 rehash，时间复杂度 O(n)，且期间整个 map 不可用。
容易踩的坑是:初始化时没有估计大小，导致容量频繁扩大。例如，已知需要保存 1000 默认构建函数(初始容量) 16，负载因子 0.75)，会触发约 6 在每次扩容之前，所有元素都必须在每次处理之前进行。

推荐显式指定初始容量：new HashMap(1024)，让 threshold = 1024 × 0.75 = 768，可容纳 768 个元素不扩容
扩展不是线程安全：多线程 put 可能会导致死循环（JDK 7)或数据丢失（JDK 必须使用8+) ConcurrentHashMap 替代

loadFactor 设太小（如 0.5)会浪费内存；设置太大(如 0.9)链表/树更容易变长，搜索慢-0.75 它是空间和时间的平衡点，一般不移动

为什么 get 操作在 JDK 8 后平均是 O(1)，但最坏的是 O(log n)
理想情况下，hash 均匀，无冲突，get 就是算 hash + 定位桶 + 直接取值，O(1)。但是一旦发生哈希冲突，就要在桶里遍历；如果是链表，最坏的是 O(n)；假如是红黑树，最坏 O(log n)。
注：这里的“最坏”并不意味着整个 map 只有一个桶，但是一个桶里有很多节点。红黑树 O(log n) 与桶内节点数相比，不是整个 map 大小。

即使开了树化，也改变不了“哈希设计差” → 冲突集中 → 单桶节点爆炸的根本问题
用 System.identityHashCode() 代替自定义 hashCode() 它不能解决商业语义上的冲突，而是改变了一种 hash 方式
调试时可用 map.size() 和 map.entrySet().stream().map(e -&gt; e.getKey().hashCode()).distinct().count() 粗略看 hash 分布是否离散

哈希冲突本身是不可消除的，只能缓解；真正决定性能上限的不是“用红黑树” key 的 hash 函数质量，数据分布特征，以及你是否在一开始就避免了明显的陷阱。
