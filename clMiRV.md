纳仪币颗犹


  
Java Base64类位于java.util.Base64提供标准、URL安全和MIME三种编码器，推荐使用，无需第三方依赖，避免使用已移除的sun.misc.BASE64Encoder。

Java 8的Base64在哪里找类，怎么用？
Java 8起，java.util.Base64不引用第三方包，是标准库自带的唯一推荐方案。分为三类:标准编码（Base64.getEncoder()）、URL安全编码（Base64.getUrlEncoder()）、MIME编码（Base64.getMimeEncoder()）。别去翻sun.misc.BASE64Encoder——那是内部API，JDK 9+已移除，运行时可直接移除NoClassDefFoundError。
常见错误现象：java.lang.NoClassDefFoundError: sun/misc/BASE64Encoder，特别是从旧项目迁移时，很容易踩到这个坑。

标准编码用于通用字符串/二进制转换，如HTTP头中的token
URL安全编码将被删除+和/换成-和_，不添加换行符，适用于URL参数或文件名
MIME编码每76个字符添加一个字符\r\n，仅在邮件或旧协议中需要，基本上不需要日常生活


encodeToString()和decode()的典型误用
最常见的错误是先把字符串起来new String(bytes)然后编码-解码失败是因为平台默认编码(如WindowsGBK)破坏了原始字节。Base64操作字节，而不是字符串。
永远基于正确的做法byte[]：用String.getBytes(StandardCharsets.UTF_8)明确指定编码，解码后使用new String(byte[], StandardCharsets.UTF_8)还原。
；
示例：


String original = &quot;hello世界&quot;;
// ✅ 正确:显式UTF-8
byte[] encoded = Base64.getEncoder().encode(original.getBytes(StandardCharsets.UTF_8));
String b64 = new String(encoded, StandardCharsets.US_ASCII);

byte[] decoded = Base64.getDecoder().decode(b64);
String restored = new String(decoded, StandardCharsets.UTF_8); // &quot;hello世界&quot;
如果跳过StandardCharsets.UTF_8，在某些环境下，中文直接变乱码，无法恢复。

流式编解码（Base64.InputStream / OutputStream）的适用场景
大文件(如几十个MB的图片，PDF）不要一次读入内存然后编码，用流式处理更稳定。但需要注意的是，这些包装流不是自动close底层流，必须手动管理；而且他们只处理“纯”base64数据，不跳过空白或换行——如果源数据包含空白或\r\n(比如从邮件正文复制的Base64)，会被抛弃IllegalArgumentException: Illegal base64 character。

适用场景：上传文件前实时编码，下载时解码磁盘
不适用场景:分析带注释或格式化的Base64文本(首先clean掉非base64字符)
性能影响:流式略慢于内存式，但内存占用从O开始(n)降至O(1)，只需要大文件

还有Apache Commons Codec或Bouncy Castle混用的风险
若项目同时存在org.apache.commons.codec.binary.Base64和java.util.Base64，不要以为只是类名不同就可以了。它们的默认行为不同：Commons CodecencodeBase64String()默认不换行，但它的Base64(true)构造器可以打开padding，JDKgetUrlEncoder()默认情况下，无padding和无关混用会导致同一段字节编写不同的结果，解码必然会失败。
建议:新代码统一使用JDK本土；旧代码迁移时，逐一检查padding是否启用。、是否忽视换行，URL是否安全，不要依靠“看起来一样”上线。
最容易被忽视的一点:JDKgetDecoder()填写末尾的字符=严格的验证，少一个会报告IllegalArgumentException；一些旧工具生成的Base64将省略padding(特别是前端JavaScript)btoa()在某些长度下不补=)，此时必须使用Base64.getDecoder().withoutPadding()配合处理。
