塘范赌讲妆


  
消耗客户端口需要使用SO_REUSEADDR/SO_REUSEPORT、扩展端口范围，启用tcp_tw_reuse；服务端需要调高ulimittit、配置limits.conf与systemd LimitNOFILE；EventlopGroup线程数应根据场景适应CPU核心数，禁止SO_LINGER，并通过ss -s定位真正的瓶颈。

客户端口不够：bind() 失败或 Address already in use

默认情况下可以建立单台客户端机器 TCP 连接数限于可用的本地端口(通常 32768–65535，共约 3.2 一万)，加上 TIME_WAIT 状态连接占据端口而不释放，压力测试时很容易遇到 Cannot assign requested address 或反复 bind() 失败。
解决方案不是堆叠机器，而是让客户端重复使用当地地址+端口，绕过端口耗尽瓶颈：

Java 客户端（如 Netty Bootstrap）调用 option(ChannelOption.SO_REUSEADDR, true) 和 option(ChannelOption.SO_REUSEPORT, true)（Linux 3.9+）
保证操作系统层面 /proc/sys/net/ipv4/ip_local_port_range 范围足够宽，例如设置为 1024 65535）
降低 TIME_WAIT 占用：设置 /proc/sys/net/ipv4/tcp_fin_timeout(不建议低于 30)，更安全的方法是启用 tcp_tw_reuse = 1(仅对客户端有效)

服务端扛不住:java.io.IOException: Too many open files

Netty 每个服务端的连接至少占用每个服务端 1 文件描述符（fd），添加日志，配置文件，JVM 实际支持的并发连接数直接局限于系统级 ulimit -n 值。默认情况通常只有 1024，连 1 万连接撑不住。
三个地方必须同时调整，缺一不可:

当前 Shell 会话：运行 ulimit -n 1048576(注:仅对当前终端和子过程生效)
系统级持久配置：编辑 /etc/security/limits.conf，添加两行：* soft nofile 1048576* hard nofile 1048576

systemd 服务（如用 systemctl 启动 Java 进程）：需在 service 显式设置在文件中 LimitNOFILE=1048576，否则 limits.conf 不生效

Netty 自身配置跟不上：EventLoopGroup 线程数与连接数不匹配
很多人认为只要调大 ulimit 万事大吉，结果 CPU 打满，连接建立变慢，甚至部分连接超时—问题经常出现 EventLoopGroup 硬件和负载模型的线程数不适合。


关键点是区分两种连接场景:

高吞吐低延迟(如 RPC）：建议 NioEventLoopGroup 线程数 ≈ CPU 核心数 × 避免过多的线程上下文切换
高连接低活动(如 IM 长连接）：线程数可以适当增加(例如) ×3～×4），但超过 64 收入急剧下降后，此时应优先考虑 GC 和内存分配
务必禁用 SO_LINGER（即不设 ChannelOption.SO_LINGER 或设为 -1），否则，关闭连接时会堵塞 EventLoop

验证是否真的是极限:不要只看连接数，要看连接数 ESTABLISHED + TIME_WAIT 分布
用 ss -s 或 netstat -s | grep -i &quot;connection&quot; 看统计比单纯数 ss -tn state established | wc -l 更可靠。细节中往往隐藏着真正的瓶颈:

如果 ss -s 显示 memory 行接近上限，说明核心 socket buffer 耗尽，需要调大 net.core.wmem_max / rmem_max

若有大量的连接卡 SYN_RECV，可能是服务端 backlog 队列溢出，需要增加 ServerBootstrap.option(ChannelOption.SO_BACKLOG, 1024) 同步调高系统 net.core.somaxconn

JVM 层面记得加 -XX:+UseG1GC -XX:MaxGCPauseMillis=50，否则 GC STW 会导致 accept 队列堆积的现象是新连接建设延迟

真正卡住的点永远不会在你预设的路径上。例如，更改后 ulimit 发现还是连不上，先 cat /proc/&lt;pid&gt;/limits&lt;/pid&gt; 确认过程的实际生效值，然后检查 dmesg | tail 看看有没有内核 OOM killer 杀戮过程的痕迹。
